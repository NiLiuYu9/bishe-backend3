# 中文注释添加 Spec

## Why
项目代码目前没有任何中文注释，作为应届毕业生和初级工程师，需要通过注释理解整个项目的业务逻辑、技术实现和代码架构。注释的目标读者是项目作者本人，注释风格为"类级 + 方法级 + 关键字段级 + 关键行内"，解释"为什么"和"业务含义"，而非重复代码。

## What Changes
- 为后端（api-platform-backend）所有 Java 文件添加中文注释
- 为前端（api-platform-frontend）所有 Vue/TS 文件添加中文注释
- 为云模块（api-platform-cloud）所有 Java 文件添加中文注释
- 注释类型包括：类/文件级注释、方法级注释、字段级注释、关键行内注释
- 不修改任何业务逻辑代码，仅添加注释

## Impact
- Affected code: 全部后端 Java 文件（~130个）、全部前端 Vue/TS 文件（~78个）、全部云模块 Java 文件（~29个）
- 不影响任何功能，纯注释添加

## ADDED Requirements

### Requirement: 后端 Java 文件中文注释
系统 SHALL 为所有后端 Java 文件添加中文注释，遵循以下标准：

#### Scenario: 类级注释
- **WHEN** 一个 Java 类被打开
- **THEN** 类声明上方应有 Javadoc 注释，说明该类的职责、所属模块、核心功能

#### Scenario: 方法级注释
- **WHEN** 一个 public/protected 方法被打开
- **THEN** 方法声明上方应有 Javadoc 注释，说明方法功能、业务流程（分步骤）、参数含义、返回值含义

#### Scenario: 字段级注释
- **WHEN** 一个 Entity/DTO/VO 字段被查看
- **THEN** 字段上方应有行注释，说明字段含义、取值范围（如状态枚举值）、冗余存储原因等

#### Scenario: 关键行内注释
- **WHEN** 代码中存在业务规则、状态流转、折扣计算等关键逻辑
- **THEN** 在该行右侧或上方添加行注释，解释"为什么这样做"和"业务规则是什么"

### Requirement: 前端 Vue/TS 文件中文注释
系统 SHALL 为所有前端 Vue/TS 文件添加中文注释，遵循以下标准：

#### Scenario: 文件级注释
- **WHEN** 一个 Vue/TS 文件被打开
- **THEN** 文件开头应有注释，说明该文件的功能、所属模块

#### Scenario: 组件方法注释
- **WHEN** 一个 Vue 组件的方法或 setup 函数被查看
- **THEN** 方法上方应有注释，说明功能、业务流程、参数含义

#### Scenario: Store/工具函数注释
- **WHEN** 一个 Pinia Store 方法或工具函数被查看
- **THEN** 方法上方应有注释，说明功能、数据流向、持久化策略等

#### Scenario: 类型定义注释
- **WHEN** 一个 TypeScript 类型/接口字段被查看
- **THEN** 字段上方应有注释，说明字段含义

### Requirement: 云模块 Java 文件中文注释
系统 SHALL 为所有云模块 Java 文件添加中文注释，遵循与后端相同的标准，并额外说明：
- 网关过滤器的执行顺序和职责
- Dubbo 服务接口的消费者方
- SDK 的使用方式

### Requirement: 注释风格一致性
所有注释 SHALL 遵循以下风格：
- 类/文件级：Javadoc 格式（Java）或 `/** */` 格式（TS），说明职责和所属模块
- 方法级：Javadoc 格式（Java）或 `/** */` 格式（TS），说明功能、流程、参数、返回值
- 字段级：`/** 字段说明 */` 格式
- 行内注释：`// 注释内容` 格式，放在关键代码行上方或右侧
- 状态值/枚举值：列出所有可能的取值及含义
- 业务流程：用分步骤说明
- 不重复代码本身，解释"为什么"和"业务含义"
